



# HelloRPA - RPA项目管理



### 一、 RPA 项目全生命周期概览



RPA项目通常遵循 **Discovery (发现) -> Design (设计) -> Develop (开发) -> Deploy (部署)** 的 4D 模型。

| **阶段**               | **核心任务**            | **关键角色**        | **交付物**                 |
| ---------------------- | ----------------------- | ------------------- | -------------------------- |
| **1. 评估与POC**       | 可行性分析、技术验证    | PM, BA, 架构师      | 评估报告, POC演示          |
| **2. 需求与分析**      | 流程梳理、编写PDD       | PM, BA, 业务SME     | **PDD (流程定义文档)**     |
| **3. 解决方案设计**    | 技术架构、编写SDD       | PM, 架构师, 开发    | **SDD (解决方案设计文档)** |
| **4. 开发与测试**      | 代码编写、单元测试、UAT | 开发, 测试, 业务SME | 代码包, UAT测试报告        |
| **5. 部署与Hypercare** | 投产、初期特护          | 运维, 开发, PM      | 运维手册, 验收报告         |

------



### 二、 关键阶段深度解析





#### 1. 概念验证 (POC - Proof of Concept)



POC不是必须的，但在面对复杂流程或新环境时非常关键。

- **目的：** 验证技术可行性（而非业务价值）。
- **何时做：**
  - 涉及非标准应用程序（如Citrix, 传统的绿屏软件, 复杂的Java应用）。
  - 涉及OCR（光学字符识别）或AI模型集成，准确率不确定时。
- **POC包含内容：**
  - 选取流程中最具技术挑战的一小部分（Happy Path）。
  - 验证机器人能否稳定识别元素、抓取数据。
- **输出：** POC演示视频或Demo，可行性确认邮件。



#### 2. 需求调研 (Process Discovery)



这是RPA项目成败的根基。作为PM，需确保BA（业务分析师）关注以下核心点：

- **流程标准化程度：** 规则是否明确？是否存在模糊的“人为判断”？(RPA最怕“凭经验”)。
- **数据结构化程度：** 输入是Excel/DB（结构化），还是扫描件/邮件正文（非结构化）？
- **系统稳定性：** 目标系统近期是否会改版？(系统UI变动会导致机器人失效)。
- **异常情况 (Exceptions)：** 必须花费50%的时间调研“如果不符合规则怎么办？”。
- **权限与环境：** 机器人是否有独立的账号？是否有测试环境？



#### 3. 核心文档：PDD 与 SDD (重中之重)



这两个文档是连接业务与技术的桥梁，也是项目验收的法律依据。



##### **A. PDD (Process Definition Document) - 流程定义文档**



- **编写者：** 业务分析师 (BA) + 业务专家 (SME)
- **阅读对象：** 业务部门、开发人员、PM
- **核心内容：**
  - **As-Is Process (现有流程)：** 详细到每一步鼠标点击、键盘输入的截图和描述。
  - **To-Be Process (目标流程)：** 引入机器人后，人与机如何协作？（例如：人处理异常，机器人处理标准件）。
  - **业务规则矩阵：** 明确的 If-Then 逻辑表。
  - **已知异常处理：** 比如“如果没有找到客户ID，机器人应该跳过还是报错？”。
  - **输入/输出规范：** 指定输入文件的格式模板。



##### **B. SDD (Solution Design Document) - 解决方案设计文档**



- **编写者：** 解决方案架构师 (SA) 或 高级开发人员
- **阅读对象：** 开发人员、运维人员
- **核心内容：**
  - **技术架构图：** 机器人、Orchestrator（控制台）、业务系统之间的连接关系。
  - **流程逻辑设计：** 详细的流程图（Flowchart），包含状态机（State Machine）设计。
  - **异常处理机制 (Error Handling)：**
    - **系统异常 (System Exception)：** 网断了、应用崩溃了（通常策略：重试3次 -> 发送告警 -> 结束）。
    - **业务异常 (Business Exception)：** 数据缺失、金额超限（通常策略：记录日志 -> 跳过该笔 -> 继续下一笔）。
  - **配置管理 (Config)：** URL、文件路径、凭证名称等如何配置（通常不写死在代码里）。
  - **日志与审计 (Logging)：** 机器人运行日志的格式和存储位置。

------



### 三、 开发、测试与交付 (Development & Delivery)





#### 1. 开发阶段 (Development)



- **代码规范：** 强制要求模块化开发（Reusable Components），避免写出巨大的“面条式代码”。
- **代码评审 (Code Review)：** 在进入测试前，架构师必须Review代码，重点看异常处理和凭证安全。



#### 2. 用户验收测试 (UAT)



RPA项目的UAT不仅是测功能，更是测**数据适应性**。

- **测试数据：** 必须使用**生产环境的脱敏数据**或**真实历史数据副本**。单纯的“造数据”往往测不出RPA的Bug。
- **签字确认：** 业务方必须签署UAT通过报告，确认机器人处理结果与人工一致。



#### 3. 部署与特护期 (Hypercare)



- **Go-Live：** 将代码包从UAT环境迁移至Production环境。
- **Hypercare (通常为1-2周)：** 此时机器人刚上线，可能会遇到未预见的生产环境问题。开发人员需处于“待命”状态，每日检查运行日志，快速修复Bug。

------



### 四、 完整的项目交付物清单 (Deliverables Checklist)



作为PM，项目结项时需确保以下文档齐全：

1. **项目章程 / 计划书** (Project Plan)
2. **流程定义文档 (PDD)** - *需业务签字*
3. **解决方案设计文档 (SDD)** - *需技术负责人签字*
4. **UAT 测试报告** (包含测试用例、测试结果) - *需业务签字*
5. **机器人操作手册 / 运维手册** (Runbook): 指导运维人员如何启动、停止机器人，以及常见报错的处理方法。
6. **代码包** (流程源码)
7. **验收确认书 (Sign-off Document)**

------



### 五、 PM 需要特别注意的 RPA 项目风险



1. **环境差异：** 测试环境和生产环境的UI、分辨率、弹窗不一致，导致上线即失败。
2. **流程变更：** 开发过程中业务流程变了（Scope Creep）。**对策：** 严格执行变更管理（Change Request），PDD签字后变更需走流程。
3. **银弹预期：** 业务方认为机器人什么都能做且100%不出错。**对策：** 提前管理预期，明确RPA是基于规则的，处理不了主观判断。
